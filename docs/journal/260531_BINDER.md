# DML Binder

It's time to think about SQL expression execution model, which involves
connecting the AST and the table values. This is kinda tricky because there are
four different things I need to connect:

- AST data type
- Table data type
- Row data value
- Runtime data value

Okay, more like three, because rows can directly return the runtime data value
that the evaluator uses. Still, there's some concerns before moving on to
implementing the binder, which itself is a question too.

- What even is a runtime data value?
- Can rows or functions return dynamic data type?
- How should a "resolved AST" and dependencies be expressed on the data model?

Also, there's some thought in the corner yelling to make a bytecode, going
"`and $0, $1; ret $0` is so beautiful, isn't it?". Some idea is valid, but
I'd rather have a recursive evaluator because otherwise I end up creating
one more representation model when I really want to have less.

## Runtime data value

But first of all, I need to figure out what "runtime data value" means first.
It'll be the basis of all runtime engine, ideally the format the row iterator
uses, which means the evaluator would also use. Currently, the row is expressed
as `std::vector<std::variant<...>>`, which works, but it may be not an ideal
format.

But then again, I can't think of a better one. This is effectively equivalent
to creating a programming language - it literally is - and the need is pretty
much equivalent to making bunch of classes going `IntegerValue`, `StringValue`,
etc, that can hold the value and additional useful information.

There are probably three approaches to do this in C++, one is going full OOP
and virtual classes - which may not be tolerable for high-performance
applications, which is the case for databases, but it's negligible for now.
One is just using `std::variant` like it's done right now.
One is COM VARIANT sadness that gives you unions and `void*`.

I think a good compromise is just keep using `std::variant` as it is preferred
for modern C++ development, but using a wrapper struct to give it a structure
to contain additional information should I have them.

For instance, something like this:

```cpp
struct NullValue {};

struct IntegerValue {
  std::int64_t value;
};

using Value = std::variant<IntegerValue, NullValue>;
```

## Bound value

Before even defining the binder, I need to figure out how I should store the
bound value, and attach it to the evaluator.

It could be done by walking and rewriting the AST, replacing the bound value
as `BoundExpr` with an index referencing the "input" array containing actual
table/row names.

```cpp
struct BoundIdentifierExpr {
  std::uint32_t index;
};

struct BoundIdentifierReference {
  std::string table_name;
  std::string column_name;
  Table * table;
  std::uint32_t index;
};

struct ExprContext {
  std::vector<BoundIdentifierReference> bound_identifiers;
};

// The processed AST should be devoid of IdentitiferExpr entirely
// Also I should do something about function calls as well..
```

This immediately makes me concerned about how the binder would be implemented,
because it needs to do a lot of jobs at once - deduping identifiers, resolving
ambiguities, etc. Unfortunately this is just a bit complicated I guess...

## Binder

So about the binder implementation. It would just walk over the AST, replace
all the `IdentifierExpr` with `BoundIdentifierExpr`. I think it would be a
good idea to implement a general purpose AST walker, so that I won't have to
repeat implementing AST walker so many times. A crude idea would be something
like this, in JS because I don't need suffering in pseudocode:

```js
// Could be used for evaluation, like this crude way, obviously terrible
walkPostorder(expr, (expr) => {
  if (expr.type === 'add') {
    assert(expr.left.type === 'numericLiteral');
    assert(expr.right.type === 'numericLiteral');
    return { type: 'numericLiteral', value: expr.left.value + expr.right.value };
  }
});
// Or
walkPostorder(expr, (expr) => {
  if (expr.type === 'identifier') {
    // Do all the binder things
    return { type: 'boundIdentifier', ... };
  }
})
```

Then the walking part gets trivial, and it's outside the scope of the binder.

So about the binder itself - it can get information about:

- What tables are involved in the `FROM`
- .... That's it

... Yes, everything must begin from resolving the table references itself,
which should be quite trivial - it just needs to point the actual table objects,
something like `std::vector<Table*>`.
Then it can begin registering the "namespace" of the tables, for instance,
`std::map<std::string, Table*>`. Alias must be used in here if specified.
Then, it can begin registering the namespace of the columns, which can be
dubious in so many circumstances. Regardless, a variable going
`std::map<std::string, std::vector<std::pair<Table*, std::uint32_t>>` would be
fine, and if any of the `IdentifierExpr` references dubious entry, an error
can be raised.

```cpp
std::vector<Table*> tables;
// An index would be a better option than dealing with raw pointers everywhere
std::map<std::string, std::uint32_t> table_namespace_map;
std::map<std::string, std::vector<std::pair<std::uint32_t, std::uint32_t>>> column_namespace_map;
```

Once that's fulfilled, pretty much all the references in anywhere can be
resolved with the map..... IF only WHERE and SELECT list existed.

Unfortunately, SQL secretly has a concept of lifetimes and whatnot - ORDER BY
can reference columns that are defined in SELECT, while simulatenously allowing
columns in some cases. So clearly I need more pondering about how this can
be implemented.
